FIRST LANGUAGE SETUP PROTOCOL

Purpose
- This protocol exists to create a personal localized copy of Signal Rail from a fresh international kit.

Hard boundary
- Do not use this protocol on a live instance.
- Do not relocalize in place.
- Do not edit the fresh international baseline in place.
- Do not localize host-project content.

Execution model
- A human or Codex-driven pass performs the localization.
- `audit_localization.ps1` validates the localized result afterward against the shipped international English baseline profile.
- The validator is not a localizer.
- The validator is not a universal language validator.
- Do not confuse semantic localization work with final validation.

Correct frame
- Input: a fresh international SR kit in a source folder
- Output: one personal localized SR core in a separate target folder

Target-copy rule
- Before any localization work, duplicate the fresh international SR kit into a separate target folder.
- Perform all edits only inside that target copy.
- The fresh international baseline kit must remain untouched.

Structural invariants
- Keep all file names unchanged.
- Keep all ID families unchanged:
  - `F-xx`
  - `D-xx`
  - `L-xx`
  - `P-xx`
  - `A-xx`
- Keep `Signal Rail` as the system name.
- Keep `AI_TO_AI__DEPLOYED_INSTANCE_SIGNAL_RAIL.txt` as the marker file name.
- Keep the canonical structure unchanged.
- Keep `--- ENTRIES START ---` and `--- ENTRIES END ---` unchanged and in English.
- Enforce these entry markers in append-driven canonicals: `04`, `05`, `08`, `09`, `97`, `98`, `99`.

What must be localized
- `00_runtime_entry.txt`
- canonical file text
- shipped lateral support file text when it belongs to the SR core, such as `97_field_findings.txt`
- AI QUICK NOTE blocks
- AI SECTION blocks
- section titles
- template labels
- form tags
- state labels
- launcher prompts and messages
- protocol files when canonical terms change

What must not be localized
- file names
- ID families
- marker file name
- `--- ENTRIES START ---` and `--- ENTRIES END ---` marker lines
- host-project content
- live project truth already written inside an active instance
- this `SR Localization Kit` support folder, unless you are explicitly rebuilding the localization tooling itself

Localization rule
- Do not translate for style first.
- Translate for protocol stability first.
- Every term must stay:
  - human-readable
  - AI-distinguishable
  - structurally consistent
  - non-overlapping with nearby concepts

Canonical work order
0. Create a separate target copy.
1. Lock the vocabulary in that target copy before editing any file.
2. Localize `00_runtime_entry.txt`.
3. Localize canonicals:
   - `01`
   - `02`
   - `03`
   - `04`
   - `05`
   - `08`
   - `09`
   - `98`
   - `99`
4. Localize shipped lateral support files:
   - `97` if it is present in the shipped baseline
5. Localize `06`.
6. Localize `07`.
7. Review the marker text if needed.
8. Localize launcher messages and bootstrap placeholders.
9. Run the final checklist.
10. Run `audit_localization.ps1` on the localized kit.

Completion condition
- The localized kit is complete only when:
  - the work was performed in a separate target copy
  - the fresh international baseline kit was not edited
  - the localized SR core is monolingual
  - no mixed labels remain
  - `06`, `07`, canonicals, and shipped lateral support files use the same structural vocabulary
  - the bootstrap still works with the localized placeholders
  - `audit_localization.ps1` returns clean for baseline-English residue and detected bootstrap/template mismatch

Short formula
- first lock the vocabulary
- then localize the kit
- then verify the bootstrap
- then freeze the personal localized copy
