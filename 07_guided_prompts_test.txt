GUIDED PROMPTS

1. Common rules
- Read 00_runtime_entry.txt first.
- Then read and follow 06_ai_to_ai.txt.
- If AI_TO_AI__DEPLOYED_INSTANCE_SIGNAL_RAIL.txt is present, treat the current folder as a deployed Signal Rail tool instance unless the user closes a different frame.
- AI_TO_AI__DEPLOYED_INSTANCE_SIGNAL_RAIL.txt does not identify the host project or the current target by default.
- Do not treat the tool surface, the working object, or a separate target under analysis as the host project by default.
- If a canonical file contains AI QUICK NOTE and AI SECTION, use the AI SECTION as the operational rule source for that file.
- Do not derive operational rules from the human section if the AI SECTION is present.
- When structuring canonical content, keep `links to` for canonical/internal references and use `external reference` for useful references outside the canonical set.
- Ask one question at a time.
- Keep each question short.
- Allow free-form answers.
- Reduce ambiguity before asking.
- Ask for the unresolved hinge, not for a full re-brief.
- Do not move on until the current point is clear enough.
- Refine the same point if the answer is partial.
- Skip what is already clear.
- If host project or mode stays unclear, stop and ask.
- Use the user's language unless asked otherwise.
- Do not infer recording language from interaction language, input format, filename, or folder alone.
- Preserve source meaning and source language until recording requires change.
- Do not turn spoken input into canonical text too early.
- In host canonical files, describe the host project by default, not Signal Rail.
- Mention Signal Rail only if it clarifies governance, current live work, or the host project is Signal Rail itself.
- If a statement exists only because of tool adoption, workflow change, or document-management setup, it does not belong in 01 by default.
- Do not infer the host project from deployed instance file presence or folder name alone.
- Do not use 03 for proposals, likely next moves, or strong ideas that are not already adopted as current live work.
- If it is not already guiding current live work, do not place it in 03.
- If content maps where the project really lives, where to enter, where not to enter, what is sensitive, or which env/deps are critical, use 08.
- Read the shape of the material before forcing destination.
- Read the stability of the material before crossing authority into a stronger canonical file.
- Be brief and concrete.
- Do not dump all questions at once.

2. Meta delivery rules
- If the user asks which prompt fits best, recommend one best fit first.
- Briefly explain what the chosen prompt is for when useful.
- When asked to provide a prompt from this file, return the original prompt verbatim and ready to use.
- Do not translate the original prompt unless the user explicitly asks.
- If translation or adaptation is requested, provide the original first and the adapted version second.

4. New project bootstrap guided
- Goal:
  - Open a new or near-blank Signal Rail instance without inventing project reality.
  - Close real use, perimeter, authority, and first live objective before writing.
- When to use:
  - When a Signal Rail instance is new, almost empty, or still mostly template, with no reliable 01-03 yet.
- 03 reconstruction note:
  - If 03 is blank or too weak, you may reconstruct a first working state from scattered sources.
  - Reconstruct the present live state, not the full project history.
  - Use temporal or historical references only when they help close what is live now.
  - If the reconstruction still branches too much, reduce the branching to the smallest live hinge; stop before writing 03 only if that hinge still cannot be closed safely.
- Boundary note:
  - Use Signal Rail governance questions to close boundaries, not to fill host identity text by default.
- Ask in this order:
  - What is the host project?

    - Example: repo name, product name, research project name.
  - Who validates project truth?

  - What is the project, in one clear description?

    - Example: a product, a protocol, a documentation system, an experiment.
  - What is Signal Rail supposed to govern here?
    - Why: This separates governance from product, implementation, and support material.

7. Review before rewrite guided
- Goal:
  - Review authority, risks, and rewrite safety before changing text.
  - Separate what is truly governed or decided from what only sounds settled.
  - Do not rewrite until the governing source is clear enough.
- When to use:
  - Before rewriting a file, section, or source set that may be cleaner than it is true.
- Hard boundary:
  - Stay in review.
- 03 reconstruction note:
  - If the rewrite target is 03 or current live state, review whether it describes the present or only reconstructs history.
  - Historical references may support a 03 rewrite, but they do not authorize turning 03 into a timeline.
  - Rewrite only the part that can be defended as current live state.
- Boundary note:
  - If reviewing 01, remove text that exists only because of Signal Rail adoption unless the host project is Signal Rail itself.
- Ask in this order:
  - What text or set of texts is under review?
    - Why: This closes the rewrite target.
    - Example: one kernel file, a working doc, a set of legacy notes.
  - Which sources may govern that target?
    - Why: This identifies live authority before cleanup.
    - Example: freeze, master, decision log, explicit owner instruction.
  - What seems already governed or decided, and what still sounds open?
    - Why: This prevents rewriting open material as if it were settled.
  - Where do you suspect conflict or distortion risk?
    - Why: This surfaces why rewrite may be dangerous.
    - Example: a cleaner doc may be less true, two sources use different meanings.
  - What outcome do you want from review?
    - Why: This prevents premature rewriting.
    - Example: identify the governing source, list risks, define rewrite-safe area.

9. Retro timeline to canonicals guided
- Goal:
  - Reconstruct a defensible project timeline before writing canonicals.
  - Separate historical reconstruction from canonical promotion.
  - Backfill Signal Rail for a project that was not governed by canonicals from day one.
- When to use:
  - The project history is short enough to reconstruct, but dense enough to create level confusion.
  - The current chat contains a lot of memory, but the canonical files were created late.
  - You need to infer identity, live state, decisions, latent material, parking, and archive from prior project evolution.
  - You do not want to write directly into canonicals from scattered chat memory.
- Hard boundary:
  - Do not write canonicals first.
  - Build the timeline first.
  - Do not invent missing history just to complete the sequence.
  - If a phase, transition, or destination is still unclear, mark it uncertain instead of forcing promotion.
- Ask in this order:
  - What is the host project?
    - Why: This closes what history is being reconstructed.
  - Which sources are valid for reconstructing the project history?
    - Why: This closes the defensible source set before interpretation.
    - Example: versioned files, notes, conversations, changelog fragments, current canonicals.
  - What are the major phases, versions, or turning points of the project?
    - Why: This creates the first timeline skeleton.
    - Example: initial mini, heavy branch, lite return, hardened line, inter-host support.
  - For each phase, what changed, why did it matter, and what did it produce?
    - Why: This separates real turning points from noise.
  - For each phase, what is its status today?
    - Why: This distinguishes live, won, latent, parked, and dead material.
    - Example: live line, winning decision, unresolved branch, useful reference, closed history.
  - Which timeline units belong to 01, 02, 03, 04, 05, 08, 98, or 99?
    - Why: This turns raw history into level-aware canonical candidates.
  - Which units are still too uncertain to promote safely?
    - Why: This prevents fake closure.
  - Do you want timeline only, timeline plus destination map, or timeline plus canonical draft proposals?
    - Why: This closes the safe output scope before writing.

  - host project:
  - mode used:
  - source set:
  - reconstructed timeline:
  - likely canonical destinations:
